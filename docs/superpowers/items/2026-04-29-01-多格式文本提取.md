# 功能项：多格式文本提取（PDF/Word/纯文本）

**来源:** [roadmap](../../docs/superpowers/plans/2026-04-29-ai-question-agent.md)
**创建:** 2026-04-29
**状态:** 进行中

## 拆解思路

当前代码库为零起点，无任何应用代码。需要从 FastAPI 最小骨架起步，再按格式复杂度递增路径构建提取能力：纯文本（native Python）→ PDF（pdfplumber）→ Word（python-docx），最后收敛为统一提取接口并补充元数据与异常处理。每个 F-N 后系统均可独立启动并通过 HTTP 验证。最简可运行的第一步是 `/health` 端点，验证 FastAPI + uvicorn + pydantic-settings 技术栈正确集成。

## 功能项

### F-1: API 服务骨架
- [x] **状态:** 完成
- **生命周期:** 交付
- **目标:** 搭建 FastAPI 最小可运行骨架，包含 `/health` 端点和 pydantic-settings 配置管理。
- **验证:**
  - `GET /health` 返回 `{"status": "ok"}` 且 HTTP 状态码 200
  - 配置文件 `.env` 中的参数可通过 `pydantic-settings` 正确加载
  - uvicorn 启动成功，OpenAPI 文档 `/docs` 可访问
- **依赖:** 无
- **代码量:** ~60 行

### F-2: 纯文本文件上传与读取
- [x] **状态:** 完成
- **验证:**
  - `POST /extract/text` 上传 UTF-8 编码的 `.txt` 文件，返回文件原始文本内容
  - 上传非文本 MIME 类型的文件返回 422 状态码
  - 上传空文件返回空字符串且状态码 200
  - 文件大小超过 50MB 时返回 413 状态码
- **依赖:** F-1
- **代码量:** ~100 行

### F-3: PDF 文本提取
- [x] **状态:** 完成
- **验证:**
  - `POST /extract/pdf` 上传中文 PDF 文件，返回按页拼接的全文文本
  - 多页 PDF 的每页文本之间以换页标记分隔
  - 无文本层的扫描版 PDF 返回空结果并附带警告信息
  - 损坏的 PDF 文件返回 422 状态码及错误描述
- **依赖:** F-1
- **代码量:** ~120 行

### F-4: Word 文本提取
- [x] **状态:** 完成
- **验证:**
  - `POST /extract/docx` 上传 `.docx` 文件，返回所有段落文本，段落间以换行分隔
  - 含表格的 `.docx` 文件同时提取表格内文本
  - 含标题样式的段落保留其样式名称作为标注
  - 非 `.docx` 格式（如 `.doc` 旧格式）上传返回 422 及明确提示
- **依赖:** F-1
- **代码量:** ~120 行

### F-5: 统一提取接口与格式自动检测
- [x] **状态:** 完成
- **验证:**
  - `POST /extract` 上传 `.txt`/`.pdf`/`.docx` 任一格式，自动识别并提取文本
  - 响应结构统一包含 `format`、`text`、`page_count` 字段
  - 无法识别的文件格式返回 422，错误信息包含支持的格式列表
  - 无扩展名但 MIME 类型可识别的文件能正确路由
- **依赖:** F-2, F-3, F-4
- **代码量:** ~150 行

### F-6: 提取结果元数据与异常处理增强
- [ ] **状态:** 待开始
- **生命周期:** 交付
- **目标:** 为统一提取接口补充字符数统计、提取耗时、警告信息等元数据；健壮化三类格式的异常处理路径。
- **验证:**
  - 响应新增 `char_count`、`extraction_time_ms`、`warnings` 字段
  - PDF 响应 `page_count` 与实际页数一致，Word 响应 `page_count` 为 null
  - 密码保护的 PDF 返回提取失败状态及具体原因
  - `.doc` 旧格式上传时 `warnings` 中有明确格式不支持提示
  - 超大文件（接近 50MB）提取耗时在合理范围内且不崩溃
- **依赖:** F-5
- **代码量:** ~130 行

### F-7: 代码清理与提取器模块化
- [ ] **状态:** 待开始
- **生命周期:** 重构
- **目标:** 移除 F-2/F-3/F-4 遗留的按格式拆分的独立端点，将三种提取逻辑收敛到 `extractors/` 模块，统一错误处理策略。
- **验证:**
  - `POST /extract/text`、`POST /extract/pdf`、`POST /extract/docx` 三个旧端点已移除
  - 仅保留 `POST /extract` 统一端点且功能不变
  - 三种提取器各自为 `extractors/` 下的独立模块，通过公共接口调用
  - 异常类统一为 `extractors/exceptions.py` 中的 `ExtractionError`
  - `GET /health` 依然返回 200
- **依赖:** F-6
- **代码量:** ~80 行
