# 功能项：章节结构识别

**来源:** [roadmap](../../docs/superpowers/plans/2026-04-29-ai-question-agent.md)
**创建:** 2026-04-30
**状态:** 完成

## 拆解思路

当前提取器只返回扁平文本，丢弃了 pdfplumber 的字号/位置和 python-docx 的 heading 样式 —— 而这些正是章节识别所需的基础数据。拆解分两条线索推进：先增强提取器输出结构化段落数据（过度），再基于该数据实现规则检测（字号突变 + 编号模式 + heading 样式映射），然后接入 GLM-5 做低置信度场景的语义兜底。规则优先覆盖 80% 常见教材排版，GLM-5 处理非标准编号体系，最后合并构建章节树并暴露统一 API。最简可运行的第一步是 `?mode=structured` 查询参数，在不动现有 36 个测试的前提下验证结构化数据可用。

## 功能项

### F-1: 结构化段落数据提取
- [x] **状态:** 完成
- **生命周期:** 过度
- **目标:** 为 `/extract` 增加 `?mode=structured` 查询参数，返回携带字体和样式元数据的段落列表，原有 `mode=text`（默认）行为不变。
- **验证:**
  - `POST /extract?mode=structured` 上传 PDF 返回的每段含 `text`、`page_number`、`font_size`、`is_bold`、`x0` 字段
  - `POST /extract?mode=structured` 上传 DOCX 返回的每段含 `style_name` 字段（如 `Heading 1`），且不再以 `[Heading 1]` 文本前缀形式出现
  - `POST /extract?mode=structured` 上传纯文本返回的每段仅含 `text` 字段，字体字段为 null
  - `POST /extract`（无 mode 参数或 `?mode=text`）行为与当前完全一致，36 个现有测试全部通过
  - PDF 提取改用 `page.extract_text_lines()` 或 `page.chars` 按行聚合，字号取段落内众数字符大小，粗体检测基于 fontname 含 `Bold` 关键字
- **依赖:** 无（直接增强现有 extractors/pdf.py 和 extractors/docx.py）
- **代码量:** ~150 行

### F-2: 基于规则的章节标题检测
- [x] **状态:** 完成
- **生命周期:** 交付
- **目标:** 消费结构化段落数据，基于编号模式匹配、字号启发、样式映射三种策略识别章节标题，输出带置信度和检测方法的标题列表。
- **验证:**
  - 检测到 `第一章 引言` 编号模式，level=1，method=pattern
  - 检测到 `1.1 背景` 编号模式，level=2，method=pattern（缩进层级由数字段数推断）
  - 检测到 `(一) 概述` 中文括号编号，level=1 或 2，method=pattern
  - DOCX 段落 `style_name=Heading 2` 直接映射 level=2，method=style，置信度 0.95
  - 无编号但字号 >= 16pt 且 is_bold=True 的段落被识别为候选标题，method=font，置信度 0.6
  - 连续多个段落符合同一 heading 特征时去重，仅保留首个
  - 纯文本文件（无字体信息）仅依赖编号模式，method=pattern
  - 检测器返回 `[{line_index, level, title, confidence, method}]` 格式
- **依赖:** F-1（消费结构化段落数据）
- **代码量:** ~140 行

### F-3: GLM-5 最小调用链路与章节语义识别
- [x] **状态:** 完成
- **生命周期:** 交付
- **目标:** 首次接入 GLM-5（zai-sdk），实现最小可用的 API 调用链路，并基于该链路实现章节语义识别 Prompt，用于处理规则无法覆盖的非标准编号教材。
- **验证:**
  - `from zai import ZhipuAiClient` 正确导入，client 可从 `settings.glm_api_key` / `settings.glm_model` 读配置初始化
  - 烟雾测试：发送含教材片段的 system + user prompt，GLM-5 返回合法 JSON（含 chapters 数组）
  - JSON mode 验证：输出 schema 含 `{title, level, start_index, end_index}` 且可被 Pydantic 校验
  - GLM-5 不可用（API key 缺失或调用超时）时返回 `None` 而非崩溃，调用方收到空结果并记录 warning
  - 章节识别 Prompt 送入一段含 `§2.3 力学定律` 或 `Module 3: Grammar` 等非中文编号的文本，GLM-5 正确识别章节边界和层级
- **依赖:** 无（独立的 LLM 模块，但逻辑上排在 F-2 后以便 F-4 合并）
- **代码量:** ~140 行

### F-4: 混合章节识别引擎
- [x] **状态:** 完成
- **生命周期:** 交付
- **目标:** 合并规则检测（F-2）和 GLM-5 语义识别（F-3）的结果，按置信度择优去重，输出统一的章节标题列表。
- **验证:**
  - 规则高置信度（>=0.8）的标题保留，低置信度区间由 GLM-5 结果填充
  - 规则和 GLM-5 对同一标题重复检测时，按 confidence 高的保留，低者丢弃
  - GLM-5 返回空（API 不可用）时引擎降级为纯规则模式，结果附带 `detection_method: rule_only` 标注
  - 规则和 GLM-5 均无结果时返回空列表，不报错
  - 合并后按 line_index 升序排列，level 序列经校验无跳跃（缺失 level 2 时自动插入占位）
- **依赖:** F-2, F-3
- **代码量:** ~130 行

### F-5: 章节树构建与统一 API
- [x] **状态:** 完成
- **生命周期:** 交付
- **目标:** 将混合引擎输出的平铺标题列表构建为父子嵌套的章节树，并通过 `POST /structure` 端点暴露完整管线（提取 + 检测 + 树构建）。
- **验证:**
  - 平铺标题 `[L1 A, L2 B, L2 C, L1 D]` 构建为 `A.children=[B, C]; D` 的嵌套树
  - 每个树节点含 `id`、`level`、`title`、`start_line`、`end_line`、`page_range`、`children`
  - `POST /structure` 上传教材 PDF/DOCX/TXT 返回完整响应：`{format, chapters: <tree>, detection_stats: {total, rule_detected, glm_detected, method}}`
  - 无章节文档返回 `chapters: null` 且 `detection_stats.total=0`，不报错
  - 超大文档（50MB/500+页）在提取阶段受现有限制保护，不会 OOM
  - `/health` 端点始终正常
  - 新增测试覆盖：中文编号教材、英文编号教材、无标题纯文本、单章文档
- **依赖:** F-1, F-4
- **代码量:** ~160 行

- **验证:**

  - extractors/ 不依赖 chapters/，保持单向依赖
  - `POST /extract?mode=structured` 的功能完全保留
  - 运行全量测试（36 个旧 + 新增章节测试）全部通过
  - ruff + mypy 检查零告警
- **依赖:** F-5
- **代码量:** ~80 行
