# progressive-plan MVP/原型驱动模式设计

**状态:** CONFIRMED
**日期:** 2026-05-01
**范围:** 修改 `.claude/skills/progressive-plan/SKILL.md`

## 背景

当前 progressive-plan 按功能层横向拆分——先把底层模块的所有 F-N 做完，再做上层。
这导致效果展示延迟、反馈循环断裂、方向风险：底层做得好不好要等上层效果来验证，
但上层要等底层全做完才能开始。

MVP/原型驱动原则：优先打通端到端可演示的垂直切片，用简化逻辑贯通多层，
尽早获取效果反馈，再根据反馈决定后续方向。

## 设计

### 1. 更新 Reasoning Process（Step 7）

当前推理步骤：
1. 评估代码现状
2. 找最小可运行步骤
3. 逐个增加能力
4. 识别过渡代码
5. 计划清理
6. 验证每步可运行

修改为：
1. 评估代码现状
2. 找最小可运行步骤
3. **识别端到端可演示切片**——从数据入口到最终用户可见输出，最短路径是什么？
   哪些模块需要硬编码/简化来打通这条路径？
4. **优先编排垂直切片**——如果功能点涉及多层（数据层→处理层→输出层），
   第一个可交付的 F-N 应该是跨层垂直切片，用简化逻辑贯通。
   后续 F-N 再逐步替换为真实实现。
5. 逐个增加能力（在垂直切片验证方向正确后，可按层深入优化）
6. 识别过渡代码和原型中的简化逻辑
7. 计划清理
8. 验证每步可运行

### 2. 新增生命周期标签 `原型`

当前标签：`交付`、`过渡`、`重构`

新增 `原型`：
- **含义**：端到端可演示的垂直切片，用硬编码/简化逻辑贯通多层
- **与 `过渡` 的区别**：`过渡` 是单层的临时实现（如内存存储→SQLite），
  `原型` 是跨层的简化贯通（如硬编码知识点→简化出题→输出题目）
- **目的**：尽早验证端到端效果，获取反馈
- **后续处理**：后续 F-N 逐步将原型中的简化逻辑替换为真实实现

### 3. 新增 `反馈` 字段

在 F-N 的验证规格中，增加可选的 `反馈` 字段：

```markdown
### F-2: 端到端出题原型 (原型)
- [ ] **状态:** 待开始
- **生命周期:** 原型
- **目标:** 用硬编码知识点 + 简化出题逻辑，打通从上传到出题的完整链路
- **验证:**
  - 上传教材 → 系统返回至少1道题目
  - 题目包含题干、选项、正确答案
- **反馈:** 🔄 完成后暂停，演示效果并评估：出题质量是否可接受？知识点粒度是否合适？
- **依赖:** F-1
- **代码量:** ~150 行
```

规则：
- 只有标记为 `原型` 的 F-N 才需要 `反馈` 字段
- dev-loop 在完成带有 `反馈` 标记的 F-N 后，暂停并提示用户演示和评估
- 评估结果可能影响后续 F-N 的方向（重新拆分、调整优先级、修改验证规格）

### 4. 更新拆解约束

在 Constraints 部分新增：
- **垂直切片优先**：如果功能点涉及 2 层及以上，至少有一个 F-N 是跨层垂直切片
  （标记为 `原型`），且应尽量靠前（F-2 或 F-3）
- **原型后有反馈检查点**：`原型` 类型的 F-N 必须包含 `反馈` 字段
- **混合策略**：不是所有 F-N 都是垂直切片——在原型验证方向正确后，
  后续 F-N 可以按层深入优化。灵活处理，根据具体功能点决定拆分策略

### 5. 更新 Good/Bad 示例

新增跨层示例：

```
Functional point: "题目生成" (current state: knowledge extraction done, no question generation)

F-1: [ ] 题目数据模型 + API 骨架 (交付) — Question model, POST /generate, ~80行
F-2: [ ] 端到端出题原型 (原型) — 硬编码知识点→简化GLM出题→返回题目, ~150行
F-3: [ ] 知识点对接 (交付) — 从/knowledge获取真实知识点, ~100行
F-4: [ ] 题型扩展 (交付) — 选择题/填空题/简答题, ~120行
F-5: [ ] 难度控制 (交付) — 难度参数调节, ~80行
F-6: [ ] 清理硬编码 (重构) — 移除原型中的硬编码知识点, ~60行
```

新增 Bad 示例：

```
F-1: 知识点模型定义 — 单层，无端到端路径
F-2: 知识点提取逻辑 — 单层，看不到出题效果
F-3: 知识点存储 — 单层，纯基础设施
F-4: 知识点层级构建 — 单层，仍看不到出题效果
F-5: 知识点API — 单层，最终用户无法感知

Why bad: 全部按层横向拆分，做到 F-5 仍然无法演示任何出题效果。
缺少垂直切片，反馈循环延迟到整个底层完成后才开始。
```

### 6. 更新输出格式模板

F-1 模板不变。F-2+ 模板增加可选字段：

```markdown
### F-N: <name>
- [ ] **状态:** 待开始
- **生命周期:** 交付 | 过渡 | 原型 | 重构
- **目标:** <!-- one sentence -->
- **验证:**
  - <!-- behavior spec 1 -->
  - <!-- behavior spec 2 -->
- **反馈:** <!-- 仅 原型 类型需要：完成后暂停，演示效果并评估什么？ -->
- **依赖:** 无 | F-<N>
- **代码量:** ~<N> 行
```

### 7. 更新 Step 8 验证清单

新增检查项：
- [ ] 如果功能点涉及 2 层及以上，至少有一个 `原型` 类型的 F-N
- [ ] 每个 `原型` 类型的 F-N 都有 `反馈` 字段
- [ ] `原型` 类型的 F-N 位置靠前（F-2 或 F-3）

## 不改的部分

- Step 1-6（输入收集、代码检查、模式选择、功能点确认）
- Step 9-11（验证、写文件、门控）
- "一次一个功能点"原则
- 5-7 个 F-N 的约束
- 文档 ≤ 200 行的约束
- 现有生命周期标签（交付、过渡、重构）的语义

## 对 dev-loop 的影响

dev-loop 在执行 F-N 时需要识别 `反馈` 字段：
- 完成 `原型` F-N 后，暂停循环，输出反馈提示
- 等待用户确认方向后再继续下一个 F-N
- 用户可能要求调整后续 F-N（重新拆分、调整优先级、修改验证规格）

这是 dev-loop 的单独改动，不在本次 progressive-plan 修改范围内，
但 progressive-plan 的输出格式需要为此提供支持（即 `反馈` 字段）。
