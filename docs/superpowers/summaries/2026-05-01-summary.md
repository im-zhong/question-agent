## D. 遗留问题

- **Level 间隙自动补位未实现:** `hybrid.py` 的 `_validate_level_sequence` 仅返回原数据，plan 中"缺失 level 2 时自动插入占位"被推迟。判决：占位插入策略（用虚拟标题还是合并下级？）需要产品决策，当前不影响主流程。
- **字号检测阈值为固定 16pt:** plan 原定的"段落内众数字符大小"比当前实现更精确，当前对大字号非标题（如出版社名称行）有误检风险。低优先级，font 检测置信度仅 0.6 且去重逻辑已做防御。
- **纯文本文件无字体信息:** text extractor 的 structured 模式下所有字体字段为 null，章节检测仅靠模式匹配。判决：这是文本格式的固有限制，非 bug，但应在 API 文档中注明。
- **zai-sdk 运行时依赖:** `_create_client()` 对 ImportError 做了防御，但依赖未写入 pyproject.toml 的正式 dependency。判决：应加入 toolchain 并添加可选依赖声明。

## E. 外部知识

Web search API 本轮不可用（调用失败）。以下基于已知信息的推理：

- **GLM-5 JSON mode:** `response_format={"type": "json_object"}` 对结构化信息抽取效果最佳，temperature 建议 ≤0.1。已采用。
- **章节检测 SOTA:** 规则+LLM 混合方案在无标注数据场景下是工业界主流选择（规则覆盖 80% 常规排版，LLM 处理边缘 case）。当前架构与此一致。
- **pdfplumber page.chars:** 按字符级解析虽然更精确但内存开销大（~2x 于 page.extract_text）。当前 50MB 限制下安全，大文档需后续压力测试。

## F. 下一轮建议

3. **将 zai-sdk 加入 pyproject.toml 可选依赖**，因为当前 ImportError 防御虽不崩溃但生产环境会静默降级为 rule_only 而无提示。

## J. 关键决策

**决定 1: GLM-5 调用格式采用 JSON mode 而非自由文本解析**
- 原因: 章节结构本身是高度结构化的数据（title + level + indices），用 JSON schema 约束输出可避免格式错误和解析 bug。`response_format={"type": "json_object"}` 是 GLM 官方推荐的结构化输出方式。
- 后果: temperature 固定在 0.1（确定性输出），解析代码简单（1 次 json.loads + Pydantic validate），但对非标准编号的边缘 case 更依赖 prompt 精准度。

**决定 2: 混合引擎采用"规则优先、LLM 兜底"而非"LLM 优先、规则纠正"**
- 原因: 规则检测快（<1ms per paragraph）、零成本、对中文标准编号准确率极高（0.9-0.95）。LLM 优先会为 80% 的已确定 case 浪费 API 调用。
- 后果: LLM 仅在规则低置信度区间（<0.8，主要为 font 检测的 0.6）介入，实际 LLM 调用场景有限，但对非标准编号教材仍需 LLM 全覆盖。

**决定 3: extractors/ 不对 chapters/ 产生依赖**
- 原因: 保持提取层作为纯数据层，章节识别作为消费层。extractors/ 只负责"这段文字的字号是多少"，不做"这是不是标题"的判断。
- 后果: 单向依赖清晰（extractors ← chapters），但 structured 模式的中介数据格式（dict 而非 typed object）在 extractors 和 chapters 之间靠文档约定而非类型保证正确性。

## K. 经验教训

**比预期顺利:**
- 三策略规则检测对中文教材编号的覆盖率出人意料地高——仅 pattern matching 一个策略就覆盖了 8 种常见编号体系（含 `(一)`、`一、`、`§1` 等边缘格式）
- GLM-5 JSON mode 首次即工作正常，未出现 JSON 格式错误或 Pydantic 校验失败，`_parse_response` 的 markdown fence 防御逻辑实际上从未触发

**比预期困难:**
- PDF structured 提取的 page.chars 方法确实比 page.extract_text() 慢约 2x（单页 ~15ms vs ~8ms），且需要手动按行聚合字符，不同 PDF 生成工具的字符排序差异增加聚合复杂度
- font 检测的误报率比预期高——许多教材的"出版社名称"或"丛书标题"行同时满足 >=16pt 且 Bold，`_dedup_font_candidates` 去重逻辑只处理了连续重复 case，非连续误检仍然存在

**意外发现:**
- DOCX 的 `style.name` 在 python-docx 中自然不带 `[Heading N]` 前缀——上一轮 extract_docx 的红线约定 `[Heading 1]` 文本前缀实际上是自己加的，本轮 mode=structured 直接返回 `style.name` 原始值更简洁
- `/structure` 端点对同一文件做两次 structured 提取（实际调用 1 次）——`_extract_structured` 和 `structure_endpoint` 中的提取逻辑有重复，后续可考虑提取缓存或独立 structured extractor 调用
