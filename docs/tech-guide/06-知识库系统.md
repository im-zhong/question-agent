# 06 - 知识库系统

知识库是本系统的"记忆"层——将文档和提取的知识点持久化存储，支持后续按需检索出题。

## 数据模型

```
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│  knowledge_bases │       │    documents     │       │ knowledge_points │
├──────────────────┤       ├──────────────────┤       ├──────────────────┤
│ id (PK, hex)     │──┐    │ id (PK, hex)     │──┐    │ id (PK, hex)     │
│ name             │  │    │ kb_id (FK)       │  │    │ document_id (FK) │
│ description      │  │    │ filename         │  │    │ kb_id (FK)       │
│ subject          │  │    │ format           │  │    │ name             │
│ grade_level      │  │    │ file_path        │  │    │ description      │
│ document_count   │  │    │ char_count       │  │    │ tags_json        │
│ created_at       │  │    │ status           │  │    │ confidence       │
│ updated_at       │  │    │ error_message    │  │    │ method           │
└──────────────────┘  │    │ created_at       │  │    │ chapter_id       │
                      │    └──────────────────┘  │    │ source_line_*    │
                      │                          │    │ created_at       │
                      │    1:N                    │    └──────────────────┘
                      └──────────────────────────┘
                      │                          1:N
                      └──────────────────────────────────┘
```

### 三层关系

```
KnowledgeBase (知识库)
  └── Document (文档)
       └── KnowledgePoint (知识点)
```

- 一个知识库可以有多个文档
- 一个文档可以提取出多个知识点
- 知识点同时冗余存储 `kb_id`，便于按知识库直接查询所有知识点

### 关键字段说明

**knowledge_bases:**
- `id`: UUID hex 字符串，如 `"a1b2c3d4e5f6..."`
- `document_count`: 冗余计数，上传文档时自增
- `subject` / `grade_level`: 可选的学科和年级标注

**documents:**
- `status`: 文档处理状态，取值为：
  - `"uploaded"` — 已上传
  - `"processing"` — 正在提取知识点
  - `"completed"` — 提取完成
  - `"failed"` — 提取失败
- `char_count`: 文档字符数

**knowledge_points:**
- `tags_json`: JSON 字符串，如 `[{"value": "加速度", "category": "concept"}]`
- `confidence`: 置信度 (0.0 - 1.0)
- `method`: 检测方法 `"rule"` / `"llm"` / `"hybrid"`
- `chapter_id`: 所属章节标识
- `source_line_start` / `source_line_end`: 原文行号范围

## 存储

### 数据库

使用 **SQLite** + **aiosqlite**，配置：

```python
PRAGMA journal_mode = WAL    # Write-Ahead Logging，支持并发读写
PRAGMA foreign_keys = ON     # 启用外键约束
```

单例连接模式，通过 `asyncio.Lock` 保证初始化安全。

### 文件存储

上传的文件保存到 `data/uploads/<kb_id>/` 目录：

```
data/
├── kb.db                        # 知识库数据库
├── chat.db                      # 对话状态数据库
└── uploads/
    ├── a1b2c3d4.../             # 知识库 ID 目录
    │   ├── 第一章.pdf
    │   └── 第二章.docx
    └── e5f6g7h8.../
        └── 习题集.txt
```

删除知识库时，对应目录通过 `shutil.rmtree` 一并删除。

## 文档处理管线

上传文档到知识库时，后端自动运行提取管线：

```
POST /v1/knowledge-bases/{kb_id}/documents
  │
  ├── 1. 保存文件到 data/uploads/<kb_id>/
  ├── 2. 创建 Document 记录 (status="uploaded")
  ├── 3. 提取文本 (extractors/)
  ├── 4. 更新 Document (status="processing", char_count)
  ├── 5. 章节检测 (chapters/hybrid)
  ├── 6. 窗口分割 (knowledge/windows)
  ├── 7. 知识点提取 (knowledge/hybrid)
  ├── 8. 批量写入 knowledge_points 表
  └── 9. 更新 Document (status="completed")

  任何步骤失败:
  └── 更新 Document (status="failed", error_message=...)
```

### 错误处理

```python
try:
    # ... 提取和知识点检测 ...
    await update_document_status(doc.id, "completed")
except Exception as e:
    await update_document_status(doc.id, "failed", str(e))
```

即使处理失败，文档记录也会保留，前端可以展示错误信息。

## 知识点检索

知识库的检索方式是**结构化查询**而非向量检索：

```python
# 按知识库 ID 获取所有知识点
SELECT * FROM knowledge_points WHERE kb_id = ? ORDER BY created_at
```

特点：
- **无嵌入**——不做文本向量化
- **无相似度搜索**——不基于语义匹配
- **全量返回**——获取知识库中的所有知识点

这种设计是刻意的：
- 知识点是结构化的（name + description + tags），不需要向量搜索
- 知识库通常不大（几篇教材），全量返回完全可行
- 避免引入向量数据库的复杂性

如果未来需要支持大规模知识库，可以在此基础上增加向量索引，但当前的"结构化 RAG-lite"已经满足需求。

## CRUD 操作

| 操作 | 方法 | 说明 |
|------|------|------|
| 创建知识库 | `POST /v1/knowledge-bases` | name 必填 |
| 列表知识库 | `GET /v1/knowledge-bases` | 按创建时间倒序 |
| 删除知识库 | `DELETE /v1/knowledge-bases/{id}` | 级联删除文档和知识点 + 文件 |
| 上传文档 | `POST /v1/knowledge-bases/{id}/documents` | 文件 + 自动提取 |
| 列表文档 | `GET /v1/knowledge-bases/{id}/documents` | 按创建时间倒序 |
| 获取知识点 | `GET /v1/knowledge-bases/{id}/knowledge-points` | 按创建时间排序 |

### 删除级联

```
DELETE /v1/knowledge-bases/{id}

1. 删除 knowledge_points WHERE kb_id = ?
2. 删除 documents WHERE kb_id = ?
3. 删除 knowledge_bases WHERE id = ?
4. 删除磁盘文件 data/uploads/{id}/
```

使用显式级联删除而非数据库外键级联，确保磁盘文件也被清理。
