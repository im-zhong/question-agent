# 04 - 题目生成引擎

题目生成是管线的最后一步——接收知识点，调用 LLM 生成结构化题目。

## 分类 Prompt 体系

这是本系统 Prompt 工程的核心设计。不是用一个通用 Prompt 出所有题目，而是根据知识点类别选择专门的 Prompt。

### 7 类 Prompt

| 类别 | 适用知识点 | 题目策略 | Prompt 关键词 |
|------|-----------|----------|--------------|
| `concept` | 概念定义 | 考定义/含义/本质特征 | "定义", "含义", "是指", "以下哪个正确描述" |
| `formula` | 公式定理 | 应用公式计算具体问题 | 需代入具体数值的场景 |
| `procedure` | 步骤流程 | 考步骤顺序/操作方法 | "步骤", "操作", "顺序" |
| `fact` | 事实数据 | 考记忆性事实 | "以下哪个", "是多少", "属于" |
| `principle` | 原理规律 | 考因果分析/逻辑推导 | "原因", "为什么", "推导" |
| `short_answer` | 用户指定简答题 | 开放式简答（1-3 句） | 需解释/计算/描述 |
| `essay` | 用户指定论述题 | 开放式论述（多步推理） | 需分析/比较/评价 |
| (fallback) | 无标签 | 通用选择题 | 直接相关即可 |

### Prompt 选择逻辑

```python
def select_prompt(tags, question_type):
    # 优先级 1: 用户明确指定题型
    if question_type in ("short_answer", "essay"):
        return QUESTION_TYPE_REGISTRY[question_type]

    # 优先级 2: 根据知识点标签类别
    if tags and tags[0]["category"] in PROMPT_REGISTRY:
        return PROMPT_REGISTRY[tags[0]["category"]]

    # 优先级 3: 通用 Prompt
    return _GENERIC_PROMPT
```

### Prompt 结构

每个 Prompt 遵循统一结构：

```
1. 角色定义
   "You are a [category] question generator for educational content."

2. 出题策略
   描述该类别应该怎么出题，使用什么关键词

3. 规则列表
   - 必须直接相关于知识点
   - 4 个选项 A/B/C/D
   - correct_answer 必须是选项标签
   - explanation 1-2 句简述
   - 仅返回 JSON

4. 输出 JSON Schema
   选择题: {stem_text, correct_answer, explanation, options[{label, text}]}
   主观题: {stem_text, reference_answer, options: []}

5. Few-shot 示例
   输入 + 输出配对
```

### 选择题 vs 主观题

```
选择题 (默认):
  输出: {
    "stem_text": "问题",
    "correct_answer": "A",
    "explanation": "解析",
    "options": [
      {"label": "A", "text": "..."},
      {"label": "B", "text": "..."},
      {"label": "C", "text": "..."},
      {"label": "D", "text": "..."}
    ]
  }

主观题 (short_answer / essay):
  输出: {
    "stem_text": "问题",
    "reference_answer": "参考答案",
    "options": []  // 空
  }
```

## 批量生成器

`questions/generator.py` 实现并发批量生成：

```python
async def generate_questions(knowledge_points, difficulty, question_type):
    # 每个知识点独立生成，使用 asyncio.to_thread 并发
    results = await asyncio.gather(*[
        asyncio.to_thread(generate_question_llm, ...)
        for kp in knowledge_points
    ])

    # 处理结果：成功 → 收录，失败 → 生成占位题干
    for idx, result in enumerate(results):
        if result is not None:
            questions.append(result)
        else:
            questions.append(_make_failed_stem(idx, kp))
```

### 并发策略

- 使用 `asyncio.to_thread` 将同步的 LLM 调用包装为协程
- `asyncio.gather` 并发执行所有知识点
- 实际并发受网络和 API 限流控制

### 容错机制

```
LLM 调用失败时:
  生成占位题干:
    stem_text: "关于「{知识点名}」的{题型}题（生成失败）"
    options: 四个占位选项 "选项A（占位）" ...
    status: "failed"
    error: "GLM-5 生成失败，返回占位题干"

设计意图:
  不让单个失败阻断整个批次
  前端可以根据 status=="failed" 做特殊展示
  用户知道哪些题目需要重新生成
```

## 难度控制

```python
class DifficultyLevel(str, Enum):
    basic = "basic"           # 基础：直接记忆/识别
    intermediate = "intermediate"  # 中等（默认）：理解/应用
    advanced = "advanced"     # 进阶：分析/综合/评价

# 难度指令作为附加 Prompt 文本
_DIFFICULTY_INSTRUCTIONS = {
    "basic": "题目应考查直接记忆和基本识别能力。",
    "intermediate": "题目应考查理解和应用能力，需要一定推理。",
    "advanced": "题目应考查分析和综合能力，需要多步推理或跨知识点综合。",
}
```

难度不是通过模型参数控制的（temperature 始终为 0.3），而是通过 Prompt 文本引导。

## LLM 调用参数

```python
# 题目生成 LLM 参数
temperature = 0.3    # 适度随机性，保证多样性但不离谱
```

## 完整数据模型

```python
class QuestionStem:
    id: int                           # 题目序号
    stem_text: str                    # 题干文本
    options: list[QuestionOption]     # 选项列表（主观题为空）
    correct_answer: str | None        # 正确答案标签（主观题为 None）
    reference_answer: str | None      # 参考答案（主观题）
    explanation: str | None           # 解析
    knowledge_point_name: str         # 关联知识点名
    question_type: str                # 题型: concept/formula/.../short_answer/essay
    difficulty: DifficultyLevel       # 难度
    status: str                       # "success" | "failed"

class QuestionOption:
    label: str   # "A" / "B" / "C" / "D"
    text: str    # 选项文本

class GenerationStats:
    total: int
    successful: int
    failed: int
```
